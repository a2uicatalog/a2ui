MCP Apps

Just launched · MCP Apps surface

This page is a minimal MCP Apps Host — the same sandboxed iframe, the same ui/initialize JSON-RPC handshake a real chat client (claude.ai, Claude Desktop) uses to render an MCP server's UI resource. Catalog atoms and hand-curated content render side by side, both mediated the same way.

What's real vs. simulated: the sandboxed iframe, the postMessage handshake, and the renderer are the genuine MCP Apps protocol and genuine catalog atom code. What's simulated is the tool call — this page feeds the view a fixed fixture payload instead of a live round-trip to a2uicatalog.ai/mcp, so you can see it work without installing an MCP client.
The launch animation isn't a catalog atom (yet) — it's gdm_rocket_panel, hand-curated content (ported from a Google Meet Stage add-on demo, stage: preview), rendered in the same surface, through the same block dispatch, right alongside the stable catalog atoms below it. That's the actual point: MCP Apps' controlled opaqueness means an agent isn't limited to what's been pre-enumerated into the catalog — curated and catalog-declarative content sit side by side in one iframe, both still only reachable through the Host's mediated channel, not a raw DOM handoff.
Connecting to view…

Handshake: the iframe (the View) posts ui/initialize to this page (the Host) on load. This page replies with a McpUiInitializeResult, the View acknowledges with ui/notifications/initialized, and this page then delivers the payload via ui/notifications/tool-result — the exact message sequence defined in the MCP Apps spec (apps.mdx). No allow-same-origin on the iframe — the View genuinely cannot reach this page's DOM.
